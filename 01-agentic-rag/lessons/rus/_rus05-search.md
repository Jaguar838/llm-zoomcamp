## Основы поиска

По своей сути любая поисковая система делает одно и то же. Она берет запрос, оценивает каждый документ на предмет сходства (similarity) и возвращает лучшие результаты.

Формально существует функция сходства:

```python
score = sim(query, document)
```

Для каждого документа в базе данных вы вычисляете эту оценку (score). Затем вы ранжируете все документы по этой оценке и возвращаете топ-N. Что отличает одну поисковую систему от другой, так это то, что именно вычисляет функция `sim`.

- Текстовый/лексический поиск (рассматривается в этом разделе): `sim` подсчитывает, сколько общих слов у запроса и документа. Он смотрит на внешнюю форму, на сами слова и ищет точные совпадения.

- Векторный/семантический поиск (рассматривается в [модуле 2](../../02-vector-search/)): `sim` сравнивает смысл запроса и документа. Та же функция, но другая мера сходства.

Рассмотрим два вопроса:

- «Can I still join the course after the start date?» (Могу ли я еще присоединиться к курсу после даты начала?)
- «Is it possible to enroll late?» (Можно ли записаться позже?)

Они означают одно и то же, но в них почти нет общих ключевых слов. «Join» — это не «enroll», слово «course» отсутствует, «start date» — это не «late». Текстовая поисковая система с трудом сопоставила бы их, потому что она видит только слова.

Позже мы увидим, как векторный поиск решает эту проблему. А пока давайте построим текстовый поиск с помощью `minsearch`.

## Индексация с помощью minsearch

У нас уже есть список `documents` из предыдущего раздела. Теперь давайте проиндексируем его.

Поиск важен, потому что у нас около 1100 документов. Отправка их всех в LLM была бы дорогой и медленной. Модель запуталась бы в таком объеме данных. Поиск позволяет находить и отправлять только самые релевантные документы.

Существует множество библиотек для поиска — Apache Lucene, Elasticsearch, Solr и другие. Но они довольно тяжеловесны. Например, для запуска Elasticsearch вам нужно запустить Docker-контейнер.

[minsearch](https://github.com/alexeygrigorev/minsearch) — это простая поисковая система, работающая в оперативной памяти. Она легковесная, поэтому работает везде, где работает Python, включая Google Colab, где нельзя запустить Docker-контейнер. Это учебная реализация, не готовая к промышленному использованию (production), но она иллюстрирует принцип работы поисковых систем и дает хорошие результаты.

Должен сделать оговорку: я написал эту библиотеку и поддерживаю ее. Она начиналась как один файл Python в первом издании LLM Zoomcamp. Я хотел показать, что поиск по ключевым словам — это не магия. Мы написали ее вместе в рамках воркшопа [Build a Search Engine](https://www.youtube.com/watch?v=nMrGK5QgPVE) (см. [код](https://github.com/alexeygrigorev/build-your-own-search-engine)).

Оказалось, что она полезна не только для обучения. За два года использования во многих проектах она зарекомендовала себя как надежный инструмент для небольших наборов данных.

Концепции в `minsearch` такие же, как в Elasticsearch (который основан на Lucene): текстовые поля (text fields), поля ключевых слов (keyword fields), бустинг (boosting), фильтрация. Я намеренно позаимствовал эти термины из Elasticsearch, так как хотел создать его легковесную замену. Так что знания, полученные здесь, пригодятся вам напрямую.

Мы проиндексируем поля `question`, `section` и `answer` как текстовые (они будут токенизированы и ранжированы), а поле `course` — как ключевое слово (для фильтрации).

Индекс токенизирует текстовые поля и обрабатывает поля ключевых слов как точные строки.

Текстовые поля — это поля, по которым вы ищете. Когда вы вводите запрос, поисковая система просматривает эти поля и токенизирует их. Она разбивает текст на слова, приводит их к нижнему регистру, удаляет стоп-слова и ранжирует результаты по релевантности. Таким образом, `question`, `section` и `answer` — это текстовые поля.

Поля ключевых слов (keyword fields) предназначены для точного совпадения. Представьте себе SQL-запрос типа `SELECT * FROM index WHERE course = 'data-engineering-zoomcamp'`. Результаты должны быть только из указанного курса, независимо от ранжирования или бустинга текстовых полей.

Вы используете поля ключевых слов, чтобы ограничить пространство поиска определенным подмножеством. В нашем случае у нас четыре курса. Допустим, вы проходите курс по LLM и задаете вопрос. Вы не хотите, чтобы ответы из курсов по MLOps или машинному обучению смешивались.

```python
from minsearch import Index

index = Index(
    text_fields=["question", "section", "answer"],
    keyword_fields=["course"]
)

index.fit(documents)
```

Вот и все, индекс построен. Название `fit` пришло из scikit-learn, где вы обучаете (fit) модель на данных. Здесь мы настраиваем индекс на наших документах.

## Пробуем поиск

Давайте попробуем выполнить поиск с вопросом, который мы использовали ранее:

```python
question = "I just discovered the course. Can I join now?"

search_results = index.search(
    question,
    boost_dict={"question": 2.0, "section": 0.5},
    filter_dict={"course": "llm-zoomcamp"},
    num_results=5
)

search_results
```

Мы получаем 5 результатов из FAQ курса LLM Zoomcamp. Лучшее совпадение — запись FAQ «I just discovered the course. Can I still join?», что нам и нужно.

Вот все вопросы из результатов:

```python
[doc["question"] for doc in search_results]
```

Мы видим вопросы о присоединении к курсу, регистрации, сертификатах и многом другом. Это документы-кандидаты, которые мы отправим в LLM.

Мы использовали `boost_dict`, чтобы придать полю `question` больший вес (2.0 вместо стандартного 1.0), а полю `section` — меньший вес (0.5). Слова запроса, появляющиеся в поле вопроса, являются более сильным сигналом, чем их появление в названии раздела.

Мы использовали `filter_dict`, чтобы возвращать результаты только из курса LLM Zoomcamp. Без этого фильтра мы получили бы результаты из всех четырех курсов.

## Бустинг полей (Boosting)

Не все поля одинаково важны. Поле `question` обычно более релевантно для поиска, чем `section`. Слова запроса в вопросе — более сильный сигнал, чем в названии раздела.

`minsearch` поддерживает бустинг полей для отражения этого:

```python
results = index.search(
    question,
    num_results=5,
    boost_dict={"question": 2.0, "section": 0.5}
)
```

Все поля имеют стандартный буст 1. Присвоение `question` буста 2 означает, что он считается в два раза важнее. Возьмем вопрос о сертификатах. Слово «сертификат» в поле вопроса теперь весит в два раза больше, чем в ответе.

Присвоение `section` значения 0.5 означает, что оно считается в два раза менее важным, так как совпадение там говорит нам меньше. Это тот же механизм бустинга, который используется в Elasticsearch и Lucene.

Попробуйте разные значения буста и посмотрите, как изменятся результаты.

## Фильтрация по курсу

Иногда вы хотите ограничить поиск конкретным курсом.

`minsearch` поддерживает фильтрацию по ключевым словам:

```python
results = index.search(
    question,
    num_results=5,
    filter_dict={"course": "mlops-zoomcamp"}
)
```

Это вернет только документы из MLOps Zoomcamp. Попробуйте несколько разных запросов и курсов, чтобы прочувствовать результаты.

```python
[doc["question"] for doc in results]
```

## Обертывание в функцию

Давайте обернем поиск в функцию `search` — первый компонент нашего конвейера RAG:

```python
def search(question, course="llm-zoomcamp"):
    boost_dict = {"question": 2.0, "section": 0.5}
    filter_dict = {"course": course}

    return index.search(
        question,
        boost_dict=boost_dict,
        filter_dict=filter_dict,
        num_results=5
    )
```

По умолчанию она ищет в FAQ курса LLM Zoomcamp.

Вы можете передать другой идентификатор (slug) курса для поиска в других курсах:

```python
search_results = search(question)
```
