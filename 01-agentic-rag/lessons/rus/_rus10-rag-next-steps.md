# Завершение части 1

Видео: [Смотреть этот урок](https://www.youtube.com/watch?v=RPXMz5p5fb8&list=PL3MmuxUbc_hLZFNgSad56pDBKK8KO0XIv)

В первой части этого модуля мы:

- Узнали, что такое RAG и почему это важно: извлечение документов, сборка промпта, генерация обоснованного ответа с помощью LLM.
- Создали поисковую систему по реальному набору данных FAQ с использованием `minsearch`.
- Создали шаблон промпта, который объединяет вопрос пользователя с результатами поиска.
- Соединили поиск, промпт и LLM в работающий конвейер RAG.
- Разделили процессы загрузки данных и выполнения запросов с помощью `sqlitesearch`.

Теперь у вас есть работающая система RAG и четкое представление о том, как взаимодействуют ее части. Дальнейшая работа заключается в улучшении каждого из этих компонентов.

## Два направления развития

[Часть 2 этого модуля: Агенты](11-agents-intro.md). Наш конвейер выполняет поиск один раз с точным запросом пользователя, без изменений. Если этот поиск возвращает «мусор», у LLM нет способа это исправить. Агент передает управление LLM. Она решает, что именно искать, сколько поисковых запросов выполнить и когда остановиться.

Агент также может обрабатывать вопросы на других языках. Он переводит запрос перед поиском, а затем переводит ответ обратно.

[Модуль 2: Векторный поиск](../../02-vector-search/). Поиск по ключевым словам ищет точные совпадения. Векторный поиск ищет по семантическому смыслу, что помогает, когда пользователь формулирует вопрос иначе, чем это сделано в FAQ.

## Elasticsearch

Elasticsearch является отраслевым стандартом для текстового поиска.

Он поддерживает:

- Полнотекстовый поиск с использованием BM25.
- Фильтрацию, агрегации и фасетный поиск.
- Векторный поиск (плотные и разреженные векторы).
- Распределенное масштабирование.
- Индексацию в реальном времени.

Он тяжеловеснее, чем `sqlitesearch`, но справляется с промышленными нагрузками любого масштаба. Если вы строите реальную систему RAG, Elasticsearch (или OpenSearch) — распространенный выбор для поискового бэкенда.

Учебное пособие по Elasticsearch можно найти в [дополнительных материалах к модулю 1](../../cohorts/2025/01-intro/elastic-search.md).

## Дообучение (Fine-tuning) против RAG

Вместо извлечения документов во время запроса можно было бы дообучить LLM на ваших данных.

Дообучение означает изменение весов модели под вашу конкретную задачу.

Это работает, но имеет свои недостатки:

- Требуется специальное оборудование (GPU) и инструменты, которые мы не рассматриваем в этом курсе.
- Модель сложно обновлять при появлении новых данных — вы же не хотите переучивать модель каждый раз, когда добавляется новый вопрос в FAQ.
- У LLM уже есть внутренние знания, но RAG дает ей доступ к информации, на которой она не обучалась.

RAG более гибок, дешев и работает с любой LLM. На практике дообучение требуется редко. Лично я никогда не сталкивался с задачей, где оно было бы действительно необходимо. Когда я анализировал около 2500 описаний вакансий для своего руководства по инженерии ИИ, лишь немногие требовали этого навыка. Поэтому сначала сосредоточьтесь на RAG, а к дообучению прибегайте только тогда, когда оно действительно необходимо.

## Что делать дальше

Попробуйте следующие шаги:

- Поэкспериментируйте с разными промптами и посмотрите, как меняются ответы.
- Добавьте больше источников данных в базу знаний.
- Попробуйте разные модели LLM (GPT-4o, Claude, Gemini, локальные модели через Ollama).
- Попробуйте Elasticsearch в качестве поискового бэкенда.
