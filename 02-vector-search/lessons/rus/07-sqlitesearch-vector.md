# Векторный поиск с sqlitesearch

Видео: [Посмотреть этот урок](https://www.youtube.com/watch?v=csxKescwJYM&list=PL3MmuxUbc_hLZFNgSad56pDBKK8KO0XIv)

В предыдущем разделе мы использовали minsearch для векторного поиска.

Он работает, но у него есть три проблемы:

- Он перестраивает индекс при каждом запуске
- Он хранит все в оперативной памяти
- Он ищет методом "грубой силы" (brute force)

При текстовом поиске мы этого не замечали. Индексация была быстрой, потому что мы ничего
не кодировали. В векторном поиске индексация запускает нейронную сеть для каждого документа,
поэтому на нашем датасете она занимает около минуты. Хранение всего в памяти здесь допустимо,
но для более крупного набора данных потребовалось бы слишком много места.

Третья проблема — поиск методом перебора. Для каждого запроса мы сравниваем вектор запроса
с вектором каждого документа. Для 1000 документов это нормально, возможно, даже быстрее,
чем любой более умный метод. Но по мере роста датасета выше 10 000 записей поиск замедляется,
и нам потребуется приближенный метод.

То, что мы делали до сих пор, — это поиск точных ближайших соседей (exact nearest neighbor, NN).
Мы оцениваем запрос относительно каждого документа и выбираем лучшие. Этот метод всегда
находит истинные лучшие совпадения, но платит за это тем, что перебирает все данные.

Приближенный поиск ближайших соседей (approximate nearest neighbor, ANN) идет по упрощенному
пути. Вместо сравнения со всем подряд он сначала сужает область поиска до региона вероятных
совпадений. Затем он ранжирует документы только внутри этого региона. Он может упустить
абсолютно лучшее совпадение, но результаты все равно хороши, а работает он гораздо быстрее.

```text
NN (точный):     сравнить запрос со ВСЕМИ документами -> топ-5
ANN (приближенный): сузить поиск до региона -> сравнить внутри региона -> топ-5
```

## sqlitesearch

sqlitesearch — это "брат" библиотеки minsearch с поддержкой постоянного хранения, и он
решает обе проблемы сразу.

Мы уже использовали его в модуле 1 для постоянного текстового поиска. Он также выполняет
векторный поиск через класс `VectorSearchIndex`. Он хранит векторы в SQLite, настоящей
базе данных на диске, и использует стратегии ANN для поиска. Поскольку данные живут на
диске, один процесс может записывать векторы, а другой — считывать их.

Если вы не установили его в предыдущем модуле, добавьте его в свой проект:

```bash
uv add sqlitesearch
```

## Создание индекса

Инициализируйте его:

```python
from sqlitesearch import VectorSearchIndex

vs_index = VectorSearchIndex(
    keyword_fields=["course"],
    mode="ivf",
    db_path="faq_vectors2.db"
)
```

sqlitesearch поддерживает три режима ANN:

- `lsh` (по умолчанию): до 100 тысяч векторов, случайные проекции гиперплоскостей
- `ivf`: от 10 до 500 тысяч векторов, кластеризация K-средних
- `hnsw`: от 10 тысяч до 1 миллиона векторов и более, граф близости (самая высокая точность)

Для нашего небольшого набора данных подойдет `lsh`. Все режимы используют двухфазный поиск:
приближенное извлечение кандидатов, затем точное переранжирование по косинусному сходству.

## Индексация данных

Обучите индекс на наших векторах и документах:

```python
vs_index.fit(vectors, documents)
```

Индекс сохраняется в файл `faq_vectors2.db`. В отличие от minsearch, этот файл остается
на диске. Вы можете выполнять поиск сразу после индексации или открыть индекс позже без
повторной индексации.

## Поиск

Поиск работает так же, как и в minsearch. Мы всегда сначала кодируем запрос в вектор.
Это одна из вещей, которая делает векторный поиск "тяжелее" текстового. В текстовом поиске
мы бы отправили сырой запрос напрямую в движок.

Закодируйте, затем выполните поиск:

```python
query = "I just discovered the course. Can I still join it?"
query_vector = model.encode(query)

results = vs_index.search(query_vector, num_results=5)
```

Посмотрите на результаты:

```python
results
```

## Фильтрация по курсу

Фильтрация работает аналогично:

```python
results = vs_index.search(
    query_vector,
    filter_dict={"course": "llm-zoomcamp"},
    num_results=5
)
```

## Закрытие соединения

Когда закончите работу с индексом:

```python
vs_index.close()
```


## Повторное открытие индекса

В новой сессии Python вы можете снова открыть индекс без пересчета эмбеддингов:

```python
from sentence_transformers import SentenceTransformer
from sqlitesearch import VectorSearchIndex

model = SentenceTransformer("all-MiniLM-L6-v2")

vs_index = VectorSearchIndex(
    keyword_fields=["course"],
    mode="ivf",
    db_path="faq_vectors2.db"
)
```

Теперь мы можем искать:

```python
query_vector = model.encode("How do I run Kafka?")
results = vs_index.search(query_vector, num_results=5)
```

Мы по-прежнему загружаем модель эмбеддингов, чтобы закодировать запрос, но мы не создаем
заново эмбеддинги для всех документов. Вызов `fit` не нужен, так как индекс уже построен
и ждет на диске.

Это то же разделение на два процесса, которое мы использовали для текстового поиска в
модуле 1. Один процесс импортирует данные и строит индекс, другой — выполняет к нему запросы.

Здесь это важнее, чем в текстовом поиске. Создание эмбеддингов для всего датасета занимает
около минуты. Мы не хотим, чтобы пользователь ждал столько времени при запуске приложения.
Мы платим эту "цену" один раз во время импорта данных, а сторона запросов запускается мгновенно.

## Использование векторного поиска sqlitesearch в RAG

Давайте используем наш постоянный векторный индекс в RAG.

В новом ноутбуке настройте модель и откройте индекс (так же, как в разделе "Повторное
открытие индекса" выше):

```python
from sentence_transformers import SentenceTransformer
from sqlitesearch import VectorSearchIndex

model = SentenceTransformer("all-MiniLM-L6-v2")

vs_index = VectorSearchIndex(
    keyword_fields=["course"],
    mode="ivf",
    db_path="faq_vectors2.db"
)
```

Мы будем использовать класс `RAGVector`, который мы определили в
[предыдущем уроке](06-rag-vector.md). Он переопределяет метод `search` для кодирования
запроса и использования векторного поиска.

Настройте клиент OpenAI и создайте ассистента:

```python
from rag_helper import RAGBase
from dotenv import load_dotenv
from openai import OpenAI

load_dotenv()
openai_client = OpenAI()

class RAGVector(RAGBase):

    def __init__(self, embedder, **kwargs):
        super().__init__(**kwargs)
        self.embedder = embedder

    def search(self, query, num_results=5):
        query_vector = self.embedder.encode(query)
        filter_dict = {"course": self.course}

        return self.index.search(
            query_vector,
            num_results=num_results,
            filter_dict=filter_dict
        )

vector_assistant = RAGVector(
    embedder=model,
    index=vs_index,
    llm_client=openai_client,
)
```

Попробуйте:

```python
vector_assistant.rag("the program has already begun, can I still sign up?")
```

Когда закончите, закройте соединение:

```python
vs_index.close()
```

## Сравнение minsearch и sqlitesearch для векторного поиска

Вот как они соотносятся:

- minsearch `VectorSearch`: в оперативной памяти (numpy), точное косинусное сходство,
  необходимо пересчитывать эмбеддинги при запуске, хорошо подходит для экспериментов
  и ноутбуков.
- sqlitesearch `VectorSearchIndex`: постоянный (файл SQLite `.db`), ANN (LSH/IVF/HNSW)
  с точным переранжированием, может открывать существующий индекс, хорошо подходит
  для проектов и обеспечения долговечности данных.

Это, вероятно, последний раз, когда вы слышите о sqlitesearch. Я создал его для обучения,
чтобы показать разделение на этап импорта и этап развертывания.

Однако у него есть и реальное применение. Его единственные зависимости — это SQLite и
numpy. Таким образом, он работает на любом хостинге, где предоставляется бесплатная база
данных SQLite, в то время как выделенная векторная база данных стоила бы дополнительных
денег. В большинстве случаев вы выберете что-то другое, к чему мы и перейдем далее.
